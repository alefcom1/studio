## Доступ по SSH

- Аутентификация — по ключу (пароль root по почте не используем).
- Ключ владельца: тип **ed25519**, комментарий `your@email.com`,
  отпечаток `SHA256:uCDAO4kx20e8DBydtvUpYgpqksxTxl6W/sK73WrcRFM`.
- Хранится на Mac владельца: `~/.ssh/sshkey` (приватный) + `~/.ssh/sshkey.pub`
  (публичный, добавлен в Hetzner). ⚠️ Изначально был создан в `/tmp` —
  перенесён в `~/.ssh/`, т.к. `/tmp` очищается при перезагрузке.
- Пассфраза ключа: пустая (можно позже переустановить `ssh-keygen -p -f ~/.ssh/sshkey`).
- Доступ подтверждён 16.07: `ssh -i ~/.ssh/sshkey root@178.105.192.76` → вход.
  Тревога Hetzner «SSH key http failure» при создании оказалась косметической
  (ключ развернулся на сервере, не сохранилась только запись в проекте).

Быстрый вход (после появления IP) — добавить в `~/.ssh/config` на Mac:

```
Host hetzner
  HostName <IP-СЕРВЕРА>
  User root
  IdentityFile ~/.ssh/sshkey
```

Тогда вход: `ssh hetzner`.

## Чеклист укрепления (по аудиту Remarka Lab) — выполнить при первой настройке

1. Обновить систему: `apt update && apt upgrade -y`.
2. Отдельный sudo-пользователь, запретить root-логин по SSH, только ключи
   (`PasswordAuthentication no`, `PermitRootLogin no` в sshd_config).
3. Файрвол (ufw): наружу только 22 (SSH), 80/443 (Caddy). БД/Redis — не публиковать.
4. Docker + Docker Compose, клон Remarka Lab, прод-профиль.
5. Caddy: обратный прокси + автоматический TLS (Let's Encrypt) на поддомене
   **`lab.remarka.biz`** (A-запись → 178.105.192.76).
6. Сменить дефолтный пароль Postgres (`sitelens_secret` в compose) на секрет из env.
7. Аутентификация приложения + мультитенантность (см. аудит — этого в коде нет).
8. Лимиты памяти контейнерам + swap (краулер может съесть память).
9. Anthropic API — прямой доступ (не нужен vercel-прокси, egress из ЕС открыт).
10. Ежедневный `pg_dump` с ротацией; healthcheck + внешний uptime-монитор.

## Планируемая архитектура: сюда же переносим remarka.biz (решение 16.07)

Владелец: перенести и сам сайт remarka.biz на этот сервер — для скорости
отдачи страниц западным посетителям (сейчас хостинг в РФ, путь
`/var/www/alefcom/...`). Обоснованно: сервер в Германии ближе к
итальянской/европейской аудитории, совпадает со стратегией «европейский рынок».

- **DNS:** `lab.remarka.biz` → 178.105.192.76 (A-запись создана 16.07).
  Позже — перенос A-записи самого `remarka.biz` на этот IP (cutover).
- **Один Caddy — два сайта:** `remarka.biz` → контейнер WordPress,
  `lab.remarka.biz` → Remarka Lab · Monitor; авто-TLS для обоих.
- **Ресурсы:** WP-витрина лёгкая (~0,5 ГБ), Remarka Lab под нагрузкой краулера
  до ~2,5–3 ГБ. На 4 ГБ + 2 ГБ swap уживаются ТОЛЬКО с memory-лимитами на
  контейнеры Remarka Lab (чтобы краулер не «убил» MySQL сайта). Рескейл до
  CX33 (8 ГБ) — 2-минутный ребут без потери данных, делаем при росте
  трафика/нагрузки. Стартуем на 4 ГБ (бюджет), масштабируемся по факту.
- **Миграция WP = чистый способ выкатить весь накопленный бэклог:** ставим
  WordPress на Hetzner свежим, импортируем тему+контент (deploy-import из
  ветки), тестируем на временном URL (напр. `new.remarka.biz`), потом cutover
  DNS. Уходим с хрупкого РФ-хостинга и его ручного flow.
- **Блокеры:** (1) deploy key сервера для приватного репо Remarka Lab;
  (2) перенос БД+медиа WordPress (dump/restore) с текущего хостинга;
  (3) SMTP (FluentSMTP) переезжает с сайтом; (4) ключи Anthropic/PSI/OAuth.

## Кабинет заказчика — план K1–K3 (в плане с 16.07, решение владельца)

Строим на базе платформы Remarka Lab (Next.js + Fastify + Prisma + BullMQ,
репо alefcom1/1russian.com) — тот же стек, тот же сервер, свой поддомен
(предложение: clienti.remarka.biz). Запуск ПОСЛЕ стабилизации
Remarka Lab · Monitor в проде.

- **K1 — фундамент (бета):** вход по magic-link (без паролей), модель
  Client → Project, изоляция данных клиента (мультитенантность), пилот на
  1–2 реальных клиентах.
- **K2 — 6 MVP-фич** (приоритет по ценности): статус проекта с этапами >
  согласования макетов/текстов (approve/comment) > обмен файлами >
  отображение счетов и статуса оплаты (bonifico/SDI — показ, не биллинг) >
  тикеты/вопросы > архив проектов.
- **K3 — после запуска сайта клиента:** дашборд метрик (PSI/CrUX, аптайм,
  позиции из Monitor), ежемесячный PDF-отчёт, e-mail-уведомления.

Синергия: кабинет и Monitor делят auth, инфраструктуру и дизайн-систему
(фирменный стиль remarka.biz — редизайн Monitor, 16.07). Задача #61 в трекере.
